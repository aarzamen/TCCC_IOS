## 3. Bugs found and fixed

### 3.1 BUG: zone numbers 1-9 were not zero-padded

**Severity:** cosmetic but spec-violating. Affects readback semantics on radio.

**Symptom:** for any coordinate in UTM zones 1-9, the encoder emitted `6V…` instead of `06V…`. Specifically broke fixtures 7, 12, and 33 in the validation set (Anchorage, antimeridian crossings).

**Root cause:** the format string in `MGRS.from()` was `"%d%@%@%@%05d%05d"` — `%d` does not zero-pad. Easting and northing were correctly `%05d`, but the zone was not.

**Why it matters operationally:**
- NGA/GeoTrans (the canonical reference) always emits 2-digit zones.
- Voice readback: "zero six victor uniform november" is the correct prowords sequence; "six victor uniform november" risks being misheard as a different zone-band combo.
- Any downstream parser (e.g. an external system displaying the LINE 1) that splits on character offsets would index into the wrong field.

**Fix applied:** changed format specifier `%d` → `%02d`. One-character change. No algorithm change. No public API change.

**Diff:**
```swift
// Before:
return String(
    format: "%d%@%@%@%05d%05d",
    zone, ...
)

// After:
return String(
    format: "%02d%@%@%@%05d%05d",
    zone, ...
)
```

The `formatted(_ compact:)` helper that splits the compact string into space-separated form already handles either width because it walks from the end (digits) backward and identifies the last 2 characters of the prefix as the square — leading zeroes flow through cleanly. Verified with new test `testFormattingPreservesLeadingZero` and `testFormattedFromCoordinateZonePadding`.

### 3.2 No other bugs found

The Snyder coefficient series, the band table, the column/row letter scheme, the Norway 32V exception, the Svalbard 31X/33X/35X/37X exception, the southern-hemisphere northing offset, and the polar rejection are **all correct**. The encoder agrees with GeoTrans to 1m on every test point — including the trickiest ones (Norway corners at exact `56.001 / 3.001` and `63.999 / 11.999`, Svalbard zone-skip gap, equatorial crossings, antimeridian).

The northing-row letter offset for even zones (the `[F..V, A..E]` cycle) is correct and verified by the Bergen / Norway fixtures (zone 32 → even → "F-offset" cycle).

The 100km-square boundary near origin (a known fragile spot, called out in the original `testOrigin` comment) is also correct: `0.0001, 0.0001` gives `31NAA66032 00011`, matching GeoTrans.

## 6. Methodology caveats

1. **Single reference implementation.** The fixtures all come from `hobuinc/mgrs`, which wraps NGA GeoTrans. While GeoTrans is the canonical implementation, we did not cross-check against `pyproj`/PROJ or against `ngageoint/mgrs-ios` directly. The Bagram and DC fixtures match the values in `RESEARCH_MGRS.md` (which were generated independently for the night-pass A2 research), giving secondary confirmation.
2. **Only forward direction tested.** The in-house encoder is forward-only by design. Round-trip parity (MGRS → lat/lon → MGRS) was not tested because the encoder doesn't decode; for the operational use case (display + voice), that's adequate.
3. **No fuzz testing.** A fuzzer that generated random `(lat, lon)` and compared to GeoTrans across millions of points would catch any pathological corner I missed. With 49 deliberately-targeted fixtures covering every boundary I could identify, my confidence is high but not absolute. Future work could wire the harness into a CI-style sweep.
4. **No 6m-band-boundary test for the GeographicLib "neighbour letter" tolerance.** Karney's library docs note that within 5 nautical miles of a band boundary, MGRS allows a neighbour letter. We didn't probe that quirk, though it shouldn't affect normal display use.

## 7. Recommendation

**The in-house MGRS encoder is FIT for 9-line MEDEVAC LINE 1 voice transmission.**

After the zone zero-padding fix:

- Every tested coordinate produces a string that is byte-identical to NGA GeoTrans output at 1m precision.
- Every documented MGRS edge case (Norway, Svalbard, latitude band boundaries, antimeridian, polar rejection, NaN rejection) is correctly handled.
- The output format conforms to NGA convention (2-digit zone, no spaces in the compact form, 5-digit easting+northing).
- The 250-LOC implementation is small enough to audit visually and stable enough that the test suite captures every important boundary.

**No further work required for the current operational scope.** A future audit could add fuzz testing against `hobuinc/mgrs` over a few thousand random points, but that's a confidence-building exercise rather than a fix.

The main risk going forward is regression — any future change to the projection math should rerun the harness (`swift /tmp/mgrs_harness.swift`) or the now-comprehensive XCTest suite (`swift test --filter MGRSTests`) before landing.
