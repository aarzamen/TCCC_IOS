# Swift TCCC

A native SwiftUI iOS app for **Tactical Combat Casualty Care** documentation. Combat medics narrate care into the device; on-device speech recognition transcribes, a deterministic state engine extracts MARCH/PAWS/vitals, and Apple's on-device language model rewrites the structured data into natural radio calls and handoff narratives.

The whole stack is **on-device only.** No telemetry, no analytics, no cloud transcription, no remote LLM.

## Architecture

```
TCCC_IOS/
├── TCCC_IOS.xcodeproj         # generated by xcodegen — never hand-edit
├── project.yml                 # source of truth for Xcode config
├── TCCC_IOS/                   # SwiftUI app target (thin)
│   ├── App/                    # AppState, lifecycle actions, demo scenarios
│   ├── Audio/                  # SpeechRecognizer (10s ring buffer + tail)
│   ├── Chrome/                 # StatusStrip, PageHeader, Panel, FooterHints
│   ├── Components/             # Cross-screen views: BigVital, ECGWave, etc.
│   ├── Design/                 # Theme tokens, typography, layout constants
│   ├── Intelligence/           # Foundation Models wrappers
│   ├── Pager/                  # 5-screen swipe pager
│   └── Screens/                # 01-05 screen views
├── Packages/TCCCKit/           # local SPM — pure logic, testable in isolation
│   └── Sources/
│       ├── TCCCDomain/         # PatientState, MARCHState, Vitals, enums
│       ├── TCCCExtractor/      # PatientStateEngine + 10 extractors + normalizer
│       ├── TCCCReports/        # 9-Line + ZMIST fallback generators
│       └── TCCCDesign/         # (placeholder for shared design tokens)
└── reference/
    ├── design_mockup/          # high-fidelity HTML/JSX prototype (not bundled)
    ├── layout_playground.html  # interactive layout-tweaking tool
    └── python_source_notes.md  # line-by-line port map
```

The app target is intentionally thin — every piece of clinical logic, every regex, every report-format helper lives in the **`TCCCKit`** package and is unit-testable without launching the simulator. The app target is just SwiftUI views + view models + iOS-specific plumbing (audio, Foundation Models, file export).

## Hard product constraints

These shape every dependency choice. If you're about to add a package, framework, or import, check it against these first:

1. **RF Ghost** — no Wi-Fi, Bluetooth, Cellular, UWB, NFC. No analytics, no crash reporting, no telemetry, no auto-update SDKs. ASR uses `requiresOnDeviceRecognition = true`. Foundation Models is on-device by design.
2. **Landscape-only, iPhone-only**, locked in `project.yml`.
3. **AES-256 at rest** via `NSFileProtectionComplete` on the data directory.
4. **Gloved-hand input** — minimum hit target 44pt for primary actions; long-press only for destructive actions with visual progress fill.
5. **NVG-safe default theme** (Tactical: deep near-black, desaturated red). Three operator-selectable themes, no auto-switch.

## Build

```bash
# regenerate the .xcodeproj from project.yml
xcodegen generate

# build for simulator
xcodebuild -project TCCC_IOS.xcodeproj -scheme TCCC_IOS \
  -destination 'generic/platform=iOS Simulator' \
  -configuration Debug build CODE_SIGNING_ALLOWED=NO

# run kit unit tests in isolation (faster)
cd Packages/TCCCKit && swift test
```

For device install (requires Apple Developer team ID set in `project.yml`):

```bash
xcodebuild -project TCCC_IOS.xcodeproj -scheme TCCC_IOS \
  -destination 'generic/platform=iOS' \
  -configuration Debug build \
  CODE_SIGNING_ALLOWED=YES \
  CODE_SIGN_STYLE=Automatic \
  DEVELOPMENT_TEAM=<your-team-id>

xcrun devicectl device install app --device <device-uuid> \
  ~/Library/Developer/Xcode/DerivedData/TCCC_IOS-*/Build/Products/Debug-iphoneos/TCCC_IOS.app

xcrun devicectl device process launch --device <device-uuid> com.aarzamen.TCCCai
```

Get the device UUID via `xcrun devicectl list devices` after plugging in.

## Demo path

1. Live Capture (default screen) — tap **Load demo · GSW thigh** to seed transcript + engine state without speaking
2. The **EXTRACTED** panel populates with ~12 facts — MOI, hemorrhage location, vitals, classification, etc.
3. Swipe through all 5 screens — each renders real engine output
4. Handoff → tap **NARRATIVE** and **ZMIST** to generate prose summaries via the on-device language model
5. Tap **JSON Encounter** or **Audio + Transcript** in the export column → iOS share sheet
6. Hold **TRANSMIT** for 2 seconds with QR · OFFLINE selected → real QR code with the patient JSON, ready to AirDrop or save
7. Footer **NEW** / **END** / **WIPE** for casualty lifecycle (single tap → big confirmation banner at top)
